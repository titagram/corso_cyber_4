Capitolo 3: Vulnerability Assessment & Triage

Slide 3.5: Proof of Concept Sicuro
Teoria: Dimostrare senza Distruggere

Un Proof of Concept (PoC) sicuro dimostra che una vulnerabilità esiste con il minimo impatto possibile.


Regole d'oro del PoC:

  • Non modificare dati in produzione
  • Non esfiltrare dati sensibili reali
  • Usare payload innocui (alert, whoami)
  • Documentare ogni step con screenshot
  • Rendere il test riproducibile dal cliente

Esempi di PoC sicuri:

  • XSS: alert('XSS') invece di cookie stealer
  • SQLi: SLEEP(5) invece di data dump
  • RCE: id; hostname invece di reverse shell

Security Insight: Un PoC ben fatto è la differenza tra un pentester professionista e un attaccante. Il nostro obiettivo è dimostrare, non distruggere.

Lab: Esempi di PoC

PoC per XSS Riflessa:

URL Malevolo: http://vulnerable-wp.local/search?q=<script>alert('XSS-PoC-2024')</script> Risultato atteso: Pop-up con messaggio 'XSS-PoC-2024'

PoC per SQL Injection (Time-based):

http://vulnerable-wp.local/page?id=1' AND SLEEP(5)-- - Risultato atteso: Risposta ritardata di 5 secondi

PoC per File Upload (RCE):

# Creiamo un file PHP innocuo └─$ echo '<?php echo "PoC-".php_uname(); ?>' > poc.php # Lo carichiamo tramite la vulnerabilità # Poi lo visitiamo per verificare l'esecuzione

✓ Dimostra RCE senza danni
